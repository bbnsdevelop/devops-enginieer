#cluster high availability on-premise
High-availability (HA) on-premise servers ensure continuous service, often targeting 99.99% or higher uptime, 
by eliminating single points of failure through redundancy. Key strategies include clustering, active-passive failover, 
load balancing, and data replication across multiple hosts to handle hardware or software failures seamlessly. 

Key Components and Strategies
Clustering & Failover: A primary server handles operations, with a standby server automatically taking over during downtime.
Redundancy: Duplicate hardware, networking, and power sources are critical to prevent localized failures from causing outages.
Data Replication: Periodic or real-time replication ensures data consistency between primary and standby servers.
Load Balancing: Distributes traffic across multiple servers, common in Remote Desktop Services (RDS).
Virtualization: Using virtual machines (VMs) facilitates faster recovery and resource reallocation. 

Implementation Best Practices
Planning: Define required uptime (SLAs), Recovery Time Objectives (RTO), and Recovery Point Objectives (RPO).
Architecture: Implement multiple nodes/servers to ensure that if one fails, others in the cluster continue to function.
Monitoring: Use tools to detect failures immediately and trigger automated failover processes.
Scope: Ensure redundancy covers not just servers, but networking, storage, and application layers

#Terraform create cluster

Step 1 — Update the system

First update Ubuntu packages.

sudo apt update
sudo apt upgrade -y
Step 2 — Install required dependencies

Install tools required to add external repositories.

sudo apt install -y gnupg software-properties-common curl
Step 3 — Add the HashiCorp GPG key

This key verifies Terraform packages.

wget -O- https://apt.releases.hashicorp.com/gpg \
| sudo gpg --dearmor \
-o /usr/share/keyrings/hashicorp-archive-keyring.gpg

This downloads the signing key used by HashiCorp packages.

Step 4 — Add the HashiCorp repository
echo "deb [arch=$(dpkg --print-architecture) \
signed-by=/usr/share/keyrings/hashicorp-archive-keyring.gpg] \
https://apt.releases.hashicorp.com \
$(lsb_release -cs) main" \
| sudo tee /etc/apt/sources.list.d/hashicorp.list

This tells Ubuntu where to download Terraform packages.

Step 5 — Update apt again
sudo apt update

This loads the new repository.

Step 6 — Install Terraform
sudo apt install terraform -y

Ubuntu will download and install Terraform.

Step 7 — Verify the installation

Check the installed version:

terraform -version

Example output:

Terraform v1.14.5
on linux_amd64

This confirms Terraform is installed successfully.

Step 8 — (Optional) Enable autocomplete

Enable CLI autocomplete:

terraform -install-autocomplete

Restart the shell afterward.

# Namespace


What is a Namespace in Kubernetes

A namespace is a logical partition inside a cluster.

It lets you divide a single cluster into multiple virtual clusters.

🎯 What namespaces actually do
1. Resource isolation (soft isolation)

Namespaces separate resources like:

Pods

Services

ConfigMaps

Secrets

Deployments

Example:

kubectl get pods -n dev
kubectl get pods -n prod

Same cluster → different environments.

2. Name scoping

Inside a namespace, resource names must be unique.

But across namespaces, they can repeat.

Example:

dev namespace → api-pod
prod namespace → api-pod

Both are valid.

3. Access control (RBAC)

You can control who can access what per namespace.

Example:

Dev team → only dev namespace

Ops team → prod namespace

Using:

Roles

RoleBindings

4. Resource quotas & limits

You can limit usage per namespace:

CPU

Memory

Number of pods

Example:

resources:
  limits.cpu: "4"
  limits.memory: "8Gi"

This prevents one team from consuming the whole cluster.

5. Network policies (security)

Namespaces help define network isolation.

Example:

Pods in dev cannot talk to prod

(using NetworkPolicy)

6. Multi-tenancy (very important)

Namespaces allow multiple teams/projects to share one cluster safely.

Think:

Cluster
 ├── dev team
 ├── QA team
 ├── production
 └── monitoring
🚫 What namespaces do NOT do

Important limitations:

❌ They do NOT isolate:

Nodes

Kernel

Network interfaces (by default)

👉 It’s logical isolation, not full isolation like VMs.

📦 Default namespaces

Kubernetes already has some:

default → where resources go if not specified

kube-system → system components (DNS, scheduler, etc.)

kube-public → publicly readable

kube-node-lease → node heartbeats

🧩 Example workflow

Create namespace:

kubectl create namespace dev

Run pod in it:

kubectl run nginx --image=nginx -n dev
🔥 Real-world DevOps usage

Namespaces are used to separate:

dev
staging
prod
monitoring
logging
ci-cd
⚖️ Namespace vs Cluster
Feature	Namespace	Cluster
Isolation	Logical	Physical/strong
Cost	Low	High
Setup	Fast	Complex
Use case	Multi-team apps	Full isolation
🧠 Mental model

Think of namespaces like:

👉 folders inside a cluster

cluster/
 ├── dev/
 ├── prod/
 └── monitoring/

Same infrastructure, organized and controlled.

🚀 Pro tip (very useful)

Always set a default namespace in your context:

kubectl config set-context --current --namespace=dev

So you don’t need -n dev every time.

✅ Summary

Namespaces:

Organize resources

Provide soft isolation

Enable RBAC

Control resource usage

Support multi-tenancy

# Summary about this issue:
🧩 Issue: kubectl logs returning 404 in Vagrant-based Kubernetes cluster
📌 Problem

When running:

kubectl logs -n dev nginx

We received:

Error from server (NotFound): the server could not find the requested resource (pods/log nginx)

Even though the pod existed and was in Running state.

🔍 Root Cause

This issue is common when using Vagrant-based clusters.

Nodes had multiple network interfaces:

NAT interface → 10.0.2.15

Private network → 172.89.x.x

The kubelet automatically picked the wrong IP (NAT)

Result:

Kubernetes API Server tried to reach kubelet via 10.0.2.15

This IP is not reachable between nodes

Logs endpoint (pods/log) failed → 404 NotFound

⚙️ Why it happens

kubelet uses:

--node-ip

If not set:

It defaults to the first interface (usually NAT in Vagrant)

✅ Solution

Explicitly configure the correct node IP for each worker.

✔️ Correct approach (persistent and recommended)

Edit:

/etc/default/kubelet

Add:

KUBELET_EXTRA_ARGS="--node-ip=<NODE_PRIVATE_IP>"

Example:

worker-1

KUBELET_EXTRA_ARGS="--node-ip=172.89.0.21"

worker-2

KUBELET_EXTRA_ARGS="--node-ip=172.89.0.22"
🔄 Apply changes
systemctl daemon-reexec
systemctl daemon-reload
systemctl restart kubelet
⚠️ Important Detail (Why previous attempt failed)

You tried modifying:

/usr/lib/systemd/system/kubelet.service.d/10-kubeadm.conf

👉 But this file is overridden by kubeadm-managed configs, especially:

/var/lib/kubelet/kubeadm-flags.env
/etc/default/kubelet   ✅ (correct override location)

👉 Therefore:

Changes in 10-kubeadm.conf may not take effect reliably

/etc/default/kubelet is the intended override layer

✅ Verification

Check node IPs:

kubectl get nodes -o wide

Before:

worker-1   INTERNAL-IP: 10.0.2.15 ❌

After:

worker-1   INTERNAL-IP: 172.89.0.21 ✅
🎯 Result

API Server can now reach kubelet

kubectl logs works correctly

Logs endpoint returns 200 OK

🧠 Final Note

If logs are empty after fix:

👉 This is expected behavior, not an error
Containers only produce logs when the application writes to stdout/stderr (e.g., nginx needs traffic).

💡 Best Practice

For kubeadm clusters, always configure node IP via:

/etc/default/kubelet (quick fix)
or

kubeadm config → nodeRegistration.kubeletExtraArgs (ideal for automation)

🧩 Problem Summary — Pods can’t communicate across nodes (Vagrant + Flannel)
🔴 Symptoms

Pods only communicate within the same node

Cross-node communication fails:

❌ ping between pods on different nodes → fails

❌ nslookup kubernetes → timeout

❌ apk add / internet access → DNS errors

Example:

DNS: transient error (try again later)
ping: bad address 'google.com'

But:

✅ Pod → same node works

❌ Pod → different node fails

🎯 Root Cause

This is a classic Vagrant + Flannel issue:

Vagrant creates multiple network interfaces:

eth0 → NAT (10.0.2.15)

enp0s8 → private network (172.89.x.x)

By default, Flannel picks the wrong interface (NAT)

👉 Result:

Pod network is built on 10.0.2.x (NAT)

Nodes cannot route traffic properly between each other

Overlay network breaks

🔥 Why it breaks everything

Flannel uses the wrong IP for:

Pod routing

VXLAN encapsulation

kube-dns (CoreDNS) becomes unreachable across nodes

API → kubelet communication may also fail indirectly

✅ Solution

Force Flannel to use the correct interface (enp0s8)

🔧 Step-by-step Fix
1. Edit Flannel DaemonSet
kubectl edit ds -n kube-flannel kube-flannel-ds
2. Add the correct interface

Inside container args:

- --ip-masq
- --kube-subnet-mgr
- --iface=enp0s8
3. Save → Flannel restarts automatically
kubectl get pods -n kube-flannel

New pods will be created with correct config.

4. Verify
kubectl describe pod -n kube-flannel <pod>

Check:

--iface=enp0s8
🧪 Validation
Before fix:

❌ DNS broken

❌ Cross-node communication fails

After fix:

✅ apk add curl works

✅ nslookup google.com works

✅ Cross-node pod communication works

✅ Cluster DNS (10.96.0.10) responds

Example:

nslookup google.com
Server: 10.96.0.10

Name: google.com
Address: 142.x.x.x
🧠 Key Insight

👉 In Vagrant environments:

Always force network interface for CNI plugins

Otherwise, they may pick:

NAT interface ❌

instead of private cluster network ✅

📌 Final Takeaways

Problem was not DNS itself

Problem was network overlay misconfiguration

Fixing Flannel interface:

restores routing

restores DNS

restores internet access

⚡ Quick Checklist (for future)

When using Vagrant + Kubernetes:

✅ Set Flannel --iface

❌ Never rely on automatic interface detection
